## Critical Findings (RED)

### RED-1: MPS Device Detection Missing — Training Falls Back to CPU

**ID:** M-001 (training-loop-debugger) + M-002 (config-auditor)  
**Files:** `src/config.py:173`, `src/config.py:66`  
**Issue:** `HardwareConfig.device` and `VLMConfig.device` use `"cuda" if torch.cuda.is_available() else "cpu"`. On Apple Silicon (M-series), this defaults to `"cpu"` despite MPS being available. Training is 10-50x slower.  
**Fix:**
```python
device: str = (
    "mps" if torch.backends.mps.is_available()
    else "cuda" if torch.cuda.is_available()
    else "cpu"
)
```

### RED-2: pin_memory=True Crashes on MPS

**ID:** M-001 (config-auditor)  
**File:** `src/autoencoder/sae_module.py:144`  
**Issue:** `pin_memory=(device != "cpu")` sets `pin_memory=True` for `"mps"` device. PyTorch MPS does not support pinned memory → RuntimeError.  
**Fix:** `pin_memory=(device not in ["cpu", "mps"])`

### RED-3: Unpinned Core ML Dependencies

**ID:** E-002, E-003 (reproducibility-checker)  
**File:** `requirements.txt`  
**Issue:** `torch>=2.2.0`, `dictionary-learning>=0.1.0`, numpy/sklearn/scipy all unpinned. Different versions produce different numerical results, making experiments unreproducible.  
**Fix:** Pin all dependencies to exact versions.

### RED-4: Missing Python Version Specification

**ID:** E-001 (reproducibility-checker)  
**File:** `pyproject.toml`  
**Issue:** No `requires-python` constraint. Different Python versions have different float semantics and RNG behavior.  
**Fix:** Add `requires-python = ">=3.10,<3.13"` to `pyproject.toml`.

### RED-5: Missing Vocabulary Embedding Creation Script — Contamination Risk

**ID:** D-002 (leak-detector)  
**Files:** `src/00_build_vocabulary.py` (exists but incomplete?), documentation gap  
**Issue:** Pipeline expects `embeddings/text_vocab_embeddings.pt` but no code explicitly creates it from vocabulary terms only. If a developer naively runs `extract_text_embeddings()` on full report corpus, vocabulary embeddings become contaminated with training data.  
**Fix:** Ensure `src/00_build_vocabulary.py` creates vocab embeddings from `data/vocabulary.json` terms ONLY. Add validation: `assert len(vocab_embeddings) == len(vocabulary_json)`.

## High-Priority Findings (ORANGE)

### ORA-1: bfloat16 Autocast Incompatible with MPS/CPU

**ID:** M-002 (training-loop-debugger), M-005 (config-auditor)  
**File:** `src/autoencoder/sae_module.py:170`  
**Issue:** `autocast_dtype=torch.bfloat16` hardcoded. MPS and older GPUs don't fully support bfloat16 autocast.  
**Fix:** `autocast_dtype = torch.float32 if device in ["cpu", "mps"] else torch.bfloat16`

### ORA-2: normalize_activations=True on Pre-Normalized Embeddings

**ID:** M-003 (training-loop-debugger)  
**File:** `src/autoencoder/sae_module.py:171`  
**Issue:** BiomedCLIP embeddings are already L2-normalized per-sample. `normalize_activations=True` applies additional normalization inside the library, potentially causing double-normalization and slower convergence.  
**Fix:** Verify dictionary_learning library behavior; likely set `normalize_activations=False`.

### ORA-3: Infinite Hang When Dataset < batch_size

**ID:** D-001, D-002 (data-pipeline-inspector)  
**File:** `src/autoencoder/sae_module.py:115-152`  
**Issue:** `drop_last=True` with `len(embeddings) < batch_size` produces zero batches. The `batch_generator()` infinite loop then hangs indefinitely.  
**Fix:** Add validation: `if embeddings.shape[0] < batch_size: raise ValueError(...)`

### ORA-4: finish_tracking() Never Called on Exception

**ID:** E-007, E-008, E-009 (interface-contract-auditor)  
**Files:** `concept_naming.py`, `generate_explanations.py`, `stability_analysis.py`  
**Issue:** If any exception occurs before the tracking block, `finish_tracking()` never executes, leaving orphaned wandb runs.  
**Fix:** Wrap each `run()` body in `try/finally` with `finish_tracking()` in `finally`.

### ORA-5: _DEFAULTS["device"] Hardcoded to "cpu"

**ID:** M-003 (config-auditor)  
**File:** `src/autoencoder/sae_module.py:51`  
**Issue:** When `SAEManager()` is constructed without explicit config (e.g., in stability_analysis.py), it silently uses CPU regardless of hardware config.  
**Fix:** Always pass explicit device from config, or derive `_DEFAULTS["device"]` dynamically.

### ORA-6: Weak Embedding Hash in Training Manifest

**ID:** E-005 (reproducibility-checker)  
**File:** `src/autoencoder/sae_module.py:609`  
**Issue:** Only first 100 samples hashed with truncated SHA256 (16 hex chars). Two different embedding files could produce identical hashes.  
**Fix:** Hash ALL samples with full SHA256.

### ORA-7: wandb Missing Git Commit, CUDA Version, Environment

**ID:** E-004 (reproducibility-checker)  
**File:** `src/autoencoder/tracking.py:20-40`  
**Issue:** `wandb.init()` doesn't capture git commit hash, CUDA version, or environment variables needed for reproduction.  
**Fix:** Log git hash, CUDA version, and key env vars in wandb config.

### ORA-8: Dead Feature Definition Inconsistency

**ID:** E-002 (metric-validator)  
**File:** `src/autoencoder/sae_module.py:390-396`  
**Issue:** Two conflicting definitions: (1) decoder norm < 1e-8 (structural), (2) never activates on dataset (empirical). These can diverge, causing inconsistent dead feature counts across pipeline stages.  
**Fix:** Standardize to single definition; prefer empirical (never activates on evaluation set).

### ORA-9: train_sae.py Manually Constructs Config Dict

**ID:** M-006 (config-auditor)  
**File:** `src/autoencoder/train_sae.py:77-85`  
**Issue:** Manually extracts SAEConfig fields into a dict. If SAEConfig gains fields, training code silently misses them.  
**Fix:** Pass `config.sae` directly: `mgr = SAEManager(config.sae)`.

### ORA-10: LR Auto-Scale Formula Hardcoded

**ID:** M-004 (config-auditor)  
**File:** `src/autoencoder/sae_module.py:591`  
**Issue:** Constants `2e-4` and `16384` for LR auto-scaling are hardcoded, not in config.  
**Fix:** Add `lr_base` and `lr_ref_dict_size` to SAEConfig.

### ORA-11: Co-activation Clustering Feature Selection Bias

**ID:** E-006 (metric-validator)  
**File:** `src/autoencoder/stability_analysis.py:165-190`  
**Issue:** Clustering uses test-set-only active features with no documentation. Features active only on test may produce spurious correlations.  
**Fix:** Document test-set bias; consider using train set for clustering.

### ORA-12: Non-Standard Entropy Metric Without Interpretation

**ID:** E-003 (metric-validator)  
**File:** `src/autoencoder/sae_module.py:455-481`  
**Issue:** Shannon entropy of feature frequencies lacks justification. Users may misinterpret high entropy as "good" when it could indicate poor specialization.  
**Fix:** Document interpretation or replace with Gini coefficient / percentile distribution.
