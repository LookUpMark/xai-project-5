## 2. Critical Path — the contamination chain

```
F-001  prepare_split() splits np.arange(N) by random index ──┐
F-002  split_embeddings() "grouping" uses file-unique basename (no-op) ─┤
                                                             ▼
        75% of TEST shares a radiograph STUDY with TRAIN
        (1121/1494; 1103 studies span both splits; 88% of studies have ≥2 views)
                                                             │
                                                             ▼
   Every held-out metric is an upper bound on a contaminated split:
   reconstruction cosine 0.9903 · dead% · dict-util · naming scores · cross-seed Jaccard
                                                             │
F-003  Two split impls + skip-if-exists ─▶ a contaminated split silently persists;
       CLI vs notebook also yield *different* partitions (non-reproducible)
```

The single highest-leverage fix is **group-splitting by patient/study** (F-001/F-002) and **consolidating onto one split implementation** (F-003). Until that is done and the SAEs retrained, no held-out number in the project reports can be read at face value.

## 3. Findings

### 🔴 Critical (RED)

#### F-001 — Train/test split ignores patient/study grouping → 75 % test contamination
- **Severity:** RED (S0) · **Category:** data leakage · **Stage:** data split
- **Location:** `src/autoencoder/train_sae.py:60-65`
- **Agent:** leak-detector · **Orchestrator-verified:** ✅ empirical re-measurement
- **Description:** `prepare_split()` builds `indices = np.arange(len(embeddings))` and calls `sklearn.train_test_split` on raw indices with no grouping. IU X-Ray stores each radiograph as `{patient}_IM-{study}-{view}.dcm.png`; 3,852 studies yield 7,470 images (88 % of studies have multiple views). A random row split cannot keep a study in one partition.
- **Evidence (measured on the real split artifacts, not a simulation):**
  ```
  visual=7470  train=5976  test=1494   (all 7470 visual ids are UNIQUE)
  distinct studies: train=3632  test=1323  overlap=1103
  TEST samples whose study also appears in TRAIN: 1121 / 1494 = 75.0 %
  studies with >1 view: 3406/3852 (88 %); max views per study = 5
  ```
- **Impact:** The held-out test set is not held out at the patient level. The SAE trains on the frontal view while the "test" set contains the lateral view of the same study — near-identical images. Reconstruction MSE/cosine, dead/utilization metrics, concept-naming scores, and cross-seed Jaccard are all materially inflated. The previously "verified" round-trip cosine **0.9903 is an upper bound, not a clean held-out figure.**
- **Remediation:** group-split by study key (`{patient}_IM-{study}`) via `sklearn.model_selection.GroupShuffleSplit`; split the image-id sidecar in lockstep; **delete** `train_embeddings.pt`/`test_embeddings.pt`/`modality_gap.pt` and retrain all 5 seeds before re-reading any held-out metric.

#### F-002 — `utils.split_embeddings` grouping is a no-op on the standard pipeline
- **Severity:** RED (S0) · **Category:** data leakage · **Stage:** data split
- **Location:** `src/utils.py:202-220`
- **Agent:** leak-detector · **Orchestrator-verified:** ✅
- **Description:** `split_embeddings()` advertises "Group by base ID to prevent data leakage" and groups via `unique_ids = list(set(image_ids))` — but `image_ids` are per-file basenames (`Path(p).name` from `extract_embeddings.py:69`). With `augmentation.enabled=False` (the default), every basename is unique, so there are 7,470 groups of size 1 and the "leak prevention" is a complete no-op. The grouping only has effect on the augmented path.
- **Evidence:** `len(set(visual_image_ids)) == len(visual_image_ids) == 7470` (measured).
- **Impact:** The one defensive measure that exists gives a false sense of safety. Even a researcher who deliberately routes through `split_embeddings` (the notebooks do) gets zero protection on the standard pipeline.
- **Remediation:** store/derive a **patient/study** group key (not the filename) and group-split on it. See F-001.

### 🟠 High (ORANGE)

#### F-003 — Two divergent split implementations + skip-if-exists ⇒ non-reproducible & sticky
- **Severity:** ORANGE (S1) · **Category:** reproducibility / evaluation leakage
- **Location:** `src/autoencoder/train_sae.py:40-42` and `src/utils.py:189-192`
- **Agent:** leak-detector · **Orchestrator-verified:** ✅
- **Description:** Two independent split code paths (`prepare_split` = random index, used by the CLI; `split_embeddings` = basename-grouped, used only by the notebooks) write to the **same** files and **both** short-circuit with `if train_path.exists() and test_path.exists(): return`. (1) Whichever entry point runs first wins → CLI and notebooks produce different partitions from the same data+seed. (2) Even after a leak fix, the stale contaminated `.pt` files are reused unchanged. (3) The image-id sidecar is written only during a fresh split, so a pre-existing tensor set leaves `generate_explanations` with no sidecar.
- **Impact:** Metrics are not reproducible across launch methods; a fixed split silently persists until files are hand-deleted.
- **Remediation:** consolidate on one group-aware split function; record the split method + a content hash of the group keys in the manifest; add `--force-resplit`.

#### F-004 — `cosine_sim` vs `cosine` key contract break ⇒ KeyError

- **Location:** `src/autoencoder/stability_analysis.py:156` (emits `"cosine_sim"`) vs every consumer reading `["cosine"]`
- **Agent:** interface-contract-auditor · **Orchestrator-verified:** ✅ (grep)

#### F-005 — `_DEFAULTS` duplicates `SAEConfig` by hand; 5 keys are unconfigurable
- **Severity:** ORANGE (S1) · **Category:** config drift / reimplementation
- **Location:** `src/autoencoder/sae_module.py:54-73` (used at `:104`, `:575`)
- **Agent:** config-hyperparam-auditor · **Orchestrator-verified:** ✅
- **Description:** `SAEManager.__init__` does `{**_DEFAULTS, **(config or {})}`. `_DEFAULTS` hand-copies every `SAEConfig` field *plus* 5 keys with no `SAEConfig` home: `lm_name`, `layer`, `chunk_size`, `dead_threshold`, `device`. The "kept in sync with config.py" comment has no enforcement; a bare `SAEManager()` (tests/notebooks/ablation scripts) silently uses stale `_DEFAULTS`. `dead_threshold` (1e-8) and `chunk_size` (512) are magic numbers for the whole naming/stability pipeline, changeable only via a dict override.
- **Remediation:** seed `_DEFAULTS` from `dataclass_to_dict(config.sae)`; promote the 5 orphaned keys to `SAEConfig`/`HardwareConfig`.

#### F-006 — `_save_manifest` recomputes lr from project constants, not the trainer's actual lr
- **Severity:** ORANGE (S1) · **Category:** reproducibility / reimplementation
- **Location:** `src/autoencoder/sae_module.py:661-666`
- **Agent:** config-hyperparam-auditor · **Orchestrator-verified:** ✅ (manifest keys confirmed on-disk)
- **Description:** When `lr=None`, the library natively computes `self.lr = 2e-4/(dict_size/16384)**0.5` (`top_k.py:200`). `_save_manifest` independently recomputes the same thing from `config["lr_base"]`/`config["lr_ref_dict_size"]`. The two agree **today** (4e-4), but the manifest value is derived from *project* constants while the *actually-used* lr comes from *library* constants — coupled only by coincidence. A library change to its base/reference silently makes the manifest lie. (`lr_base`/`lr_ref_dict_size` exist *only* to feed this reimplementation; the library ignores them.)
- **Remediation:** read and persist `trainer.lr` (the real value); drop `lr_base`/`lr_ref_dict_size`.

#### F-007 — training manifest omits optimizer + library hyperparameters
- **Severity:** ORANGE (S1) · **Category:** reproducibility
- **Location:** `src/autoencoder/sae_module.py:668-689`
- **Agent:** reproducibility-checker + config-hyperparam-auditor · **Orchestrator-verified:** ✅ (on-disk manifest keys confirmed)
- **Description:** The manifest captures seed/steps/batch_size/lr/dict_size/k/warmup/decay/log_steps/autocast/normalize/device/embeddings/hash/torch — but **omits**: Adam `betas`/`weight_decay`/`eps`, and the TopK library's `auxk_alpha`, `dead_feature_threshold` (=10 000 000), `threshold_beta`, `threshold_start_step`, `clip_grad_norm`. These drive dead-feature dynamics and are library defaults. After a library bump, the manifest would silently misrepresent the run. (The library writes its own `trainer_0/config.json` with these, but the project treats `training_manifest.json` as canonical.)
- **Remediation:** merge the library `trainer_0/config.json` into the project manifest, or hardcode the constants into `SAEConfig`.

#### F-008 — `compute_sparsity_metrics` entropy is conditioned on the active subset; docstring's `log(dict_size)` max is wrong
- **Severity:** ORANGE (S1) · **Category:** metric validity
- **Location:** `src/autoencoder/sae_module.py:533-536`
- **Agent:** metric-validator · **Orchestrator-verified:** ✅
- **Description:**
  ```python
  freq = active_per_feature / (active_per_feature.sum() + 1e-8)
  freq = freq[freq > 0]              # <-- conditions on the active subset
  entropy = -(freq * freq.log()).sum().item()
  ```
  The entropy is computed over only the features that fire on this batch, so its true maximum is `log(n_active)`, not `log(dict_size)` as the docstring (line 516) claims. The raw number is a valid entropy over the active subset, but the documented scale is wrong, so the natural utilization ratio `entropy/log(dict_size)` mis-states how concentrated the dictionary is.
- **Remediation:** either compute over all `dict_size` features (dead as zeros, max genuinely `log(dict_size)`), or document it as "entropy over the active subset" and report `log(n_active)` as the reference max.

#### F-009 — No `use_deterministic_algorithms` / `CUBLAS_WORKSPACE_CONFIG` ⇒ bitwise repro not guaranteed (CUDA)
- **Severity:** ORANGE (S1) · **Category:** reproducibility / non-determinism
- **Location:** `src/utils.py:41-48` (`set_global_seed`); no `use_deterministic_algorithms` anywhere in `src/`
- **Agent:** reproducibility-checker · **Orchestrator-verified:** ✅ (grep — absent)
- **Description:** `set_global_seed` seeds RNGs and sets `cudnn.deterministic`, but never calls `torch.use_deterministic_algorithms(True)` nor sets `CUBLAS_WORKSPACE_CONFIG`. The TopK path uses `topk(sorted=False)` + `scatter_`, which are non-deterministic-on-CUDA. Two same-seed runs on the same GPU are therefore not bitwise-guaranteed. (On the MPS/CPU host used here the practical risk is lower, but the project's multi-seed stability claim implicitly assumes reproducibility.)
- **Remediation:** call `torch.use_deterministic_algorithms(True, warn_only=True)` in `set_global_seed`; assert `CUBLAS_WORKSPACE_CONFIG` when `device=="cuda"`.

#### F-010 — Augmentation modules use `from src.` imports ⇒ crash under `PYTHONPATH=src`
- **Severity:** ORANGE (S1) · **Category:** data pipeline / import contract
- **Location:** `xai_datasets/augmentation.py:3-4`, `src/augmentation/transforms.py:2`
- **Agent:** data-pipeline-inspector · **Orchestrator-verified:** ✅ (grep)
- **Description:** Both files use `from src.config import …` / `from src.augmentation.transforms import …`, while every other module uses bare sibling imports after putting `src/` on `sys.path` (e.g. `concept_naming.py:21` `import config`, `extract_embeddings.py:8` `from config import VLMConfig`). The augmented extraction works today only because the notebook happens to put both `PROJECT_ROOT` and `PROJECT_ROOT/src` on the path. Under the documented `PYTHONPATH=src` convention the augmented ablation raises `ModuleNotFoundError: No module named 'src'`. The augmentation path is untested (86 passing tests run with `augmentation.enabled=False`).
- **Remediation:** switch both files to bare imports (`from config import …`, `from augmentation.transforms import …`).

## 5. Investigated & Refuted (adversarial verification at work)

These candidate findings were raised by agents and **rejected by orchestrator re-verification** — included for transparency.

- **R-1 — "`drop_last=True` permanently drops the same 232 samples, never seen"** *(reproducibility, candidate ORANGE).* **Refuted.** `train()` (`sae_module.py:174-182`) passes a **seeded generator** with `shuffle=True`; `DataLoader` re-shuffles each epoch and the generator *advances* (it is not re-seeded), so the dropped partial batch differs every epoch. Over ~2,000 epochs every sample is seen many times. The code's own log comment ("step-based training cycles many epochs, so all data is seen") is correct.
- **R-2 — "`high_corr_pairs //= 2` is odd-unsafe"** *(metric, sub-point of F-013).* **Refuted.** `co_occurrence` is a symmetric matrix; after subtracting the diagonal, the off-diagonal excess is always even, so `//= 2` is exact. (The *diagonal-in-mean* sub-point survives as F-013.)

- **Data integrity:** vocabulary build is RadLex-derived and split-independent (no leak). Embeddings are L2-normalized; the image-id sidecar is row-aligned with the visual tensor. The integrity failure is **entirely at the split stage** (F-001/F-002/F-003).

- **Reproducibility:** seed propagation is complete, but bitwise reproducibility is not guaranteed (F-009/F-019) and provenance capture is incomplete (F-006/F-007/F-016/F-018).
