**Pass / Fail: FAIL (evaluation integrity).** The judge's per-concept judging loop is structurally correct and its label parser works — but **two S0 bugs silently corrupt the MedConcept headline metric `Score(c)`**, and an S1 reproducibility gap makes the whole stage non-reproducible:

1. **F-001 (S0)** — the prompt primes the verbs `SUPPORT/CONTRADICT/AMBIGUOUS` but demands the unrelated labels `Aligned/Unaligned/Uncertain` with **no verb→label mapping** (the explicit Rules block from the spec, `implementation.md:219-223`, was dropped). The parser only accepts the 3 labels (verified empirically: `_extract_verdict("Supports")→None`). Verb-shaped answers → retry → fallback `Uncertain`, **biasing `Score(Uncertain)` up and `Score(Aligned/Unaligned)` down**, and tripling LLM cost on affected pairs.
2. **F-002 (S0)** — errored pairs are persisted as `verdict="Uncertain", raw_response="ERROR: …"` and `load_checkpoint()` rebuilds `done_keys` from **all** records unconditionally, so `--resume` **never retries** them. Transient infra failures become **permanent frozen verdicts** indistinguishable from real ones in the output CSV (`_save_final` strips `raw_response`).
3. **F-003 (S1)** — the retry path uses `do_sample=True, temperature=0.3` with **no seed** anywhere (the existing `utils.set_global_seed` at `utils.py:41` is never called), so retried pairs are non-reproducible — violating the spec's `temperature=0.0` deterministic contract.

**Blocker summary:** F-001 and F-002 invalidate the integrity of `Score(c)` (the project's headline interpretability metric). Both are **cheap to fix** (small, localized edits) and require **no retraining** — unlike the prior audit's split-leakage cluster. They should be fixed and the judge re-run before any Aligned/Unaligned/Uncertain figure is reported.

## Does the judge follow MedConcept?

**Partially — not fully compliant.**

- **Logical structure: ✅ compliant.** The eval loop judges each predicted concept `p_i` against the **ground-truth** radiology report (joined via `indiana_reports.csv` + `indiana_projections.csv`), **not** the model's own `pseudo_report`. This matches the MedConcept formalism `Score(c) = (1/|P|)·Σ 1(v_i = c)` where `v_i` is the verdict on predicted concept `p_i` (`implementation.md:1156-1163`). Judging the pseudo_report against itself would be circular — the current behaviour is correct. *(Refuted candidate J10 — see §Investigated & Refuted.)*
- **Label parsing: ✅ correct.** `_extract_verdict` parses the three labels case-insensitively, exact or as a standalone token (`"aligned"→Aligned`, `"The answer is Unaligned"→Unaligned`, `"Aligned."→Aligned`).
- **Prompt→label mapping: ❌ non-compliant.** The spec's prompt (`implementation.md:215-223`) carries an explicit Rules block (`SUPPORTS→Aligned`, `CONTRADICTS→Unaligned`, `AMBIGUOUS→Uncertain`). The shipped prompt (`evaluate_llm_judge.py:49-61`) **dropped it** and primes verbs while demanding labels with no mapping → **F-001**.
- **Metric persistence & population: ❌ non-compliant.** `Score(c)` is computed **arithmeticly** inside `_print_summary` (correct formula) but (a) **never persisted** to `aligned_scores.csv` or any sidecar (F-005), and (b) its denominator `|P|` is **contaminated** by infra-error `Uncertain` records (F-002/F-004) and **silently reduced** by dropped no-report images (F-006).
- **Determinism contract: ❌ non-compliant.** Spec mandates `temperature=0.0` fully-deterministic MedGemma via Ollama (`implementation.md:190,235`); the implementation uses HF `pipeline` greedy-first + `temperature=0.3` unseeded sampling on retry (F-003).

## Critical Path — the metric-corruption chain

```
F-001  prompt primes verbs, parser has no verb→label map ──┐
        "Supports"/"Contradicts"/"Ambiguous" → None → fallback "Uncertain" ─┤
                                                                             │
F-002  exception handler saves verdict="Uncertain" raw="ERROR: …"           │
        load_checkpoint puts it in done_keys → --resume never retries ───────┤
                                                                             ▼
        Score(c) = (1/|P|) Σ 1(v_i = c)  is computed on a CONTAMINATED vector:
        │P| includes frozen infra-error "Uncertain" (F-002) + parse-fallback "Uncertain" (F-001)
        and silently EXCLUDES no-report images (F-006)
                                                  │
F-003  unseeded temperature=0.3 retry → the contaminated vector is ALSO non-reproducible ─┘
                                                  │
F-004  headline "Alignment Rate" divides Aligned by ALL records (incl. errors/fallbacks)
F-005  Score(c) never persisted → no canonical artifact, every consumer recomputes (possibly wrong)
```

The single highest-leverage fix is **F-001 + F-002** (both S, localized, no retrain). They remove the two systematic biases in the verdict vector. **F-003** then makes the (now-unbiased) verdicts reproducible.

## Findings

### 🔴 Critical (S0)

#### F-001 — Prompt primes verbs SUPPORT/CONTRADICT/AMBIGUOUS but parser only accepts labels → verb answers silently become fallback "Uncertain"
- **Severity:** S0 · **Category:** evaluation / interface · **Stage:** prompt + verdict parsing
- **Location:** `src/evaluate_llm_judge.py:49-61` (prompt) + `237-250` (`_extract_verdict`)
- **Agents:** all 9 · **Orchestrator-verified:** ✅ empirical
- **Description:** `JUDGE_PROMPT_TEMPLATE` asks *"Does the report SUPPORT, CONTRADICT, or is AMBIGUOUS about this concept?"* then demands *"Answer with exactly one word: Aligned, Unaligned, or Uncertain."* It primes three **verbs** but demands three unrelated **labels** with no stated mapping. The intended spec (`implementation.md:215-223`) carried an explicit Rules block (`SUPPORTS→Aligned`, `CONTRADICTS→Unaligned`, `AMBIGUOUS→Uncertain`) that the shipped prompt **dropped**. `_extract_verdict` (237-250) accepts **only** the three labels (exact case-insensitive, or as a standalone whitespace token); it performs **no** verb→label mapping.
- **Evidence (empirical, run in isolation):**
  ```
  _extract_verdict("Supports")    -> None
  _extract_verdict("Support")     -> None
  _extract_verdict("Contradicts") -> None
  _extract_verdict("Ambiguous")   -> None
  _extract_verdict("aligned")     -> "Aligned"   (labels work)
  ```
  When a verb slips through, `parse_and_validate` (197) increments retries; after 2 retries `route_after_validation` returns `"fallback"` → `fallback` node forces `"Uncertain"` (209). Verbs are the lexically natural one-word answer to a SUPPORT/CONTRADICT/AMBIGUOUS question, so this is a plausible **systematic** failure, not an edge case.
- **Impact:** Systematic upward bias of `Score(Uncertain)` and downward bias of `Score(Aligned)/Score(Unaligned)` for any pair where MedGemma echoes the primed verb. Silently corrupts the headline metric. Triples LLM cost on affected pairs (3 calls). No test feeds a verb-shaped answer (tests feed only the 3 labels or unrelated garbage like `"Maybe"`) → **uncaught**.
- **Remediation:** (a) Restore the Rules block from `implementation.md:219-221` into `JUDGE_PROMPT_TEMPLATE`; **and/or** (b) add a verb→label alias map in `_extract_verdict` before returning `None`: `{"support","supports","supported"}→Aligned`, `{"contradict","contradicts","contradicted","deny","denies","denied"}→Unaligned`, `{"ambiguous","ambiguity","unclear"}→Uncertain`. Add a test `assert _extract_verdict("Supports") == "Aligned"`.

#### F-002 — Errored pairs persisted as verdict="Uncertain" are treated as DONE by --resume and never retried → silent permanent failure
- **Severity:** S0 · **Category:** data · **Stage:** checkpoint / resume
- **Location:** `src/evaluate_llm_judge.py:455-465` (exception handler) + `257-263` (`load_checkpoint`) + `402` (skip)
- **Agents:** all 9 · **Orchestrator-verified:** ✅
- **Description:** When `judge.invoke()` raises, the `except` block (455-465) appends `{verdict: "Uncertain", raw_response: "ERROR: {e}"}` to `records`, which `save_checkpoint` writes to `CHECKPOINT_PATH` (468-469, 474). On the next `--resume`, `load_checkpoint()` (257-263) rebuilds `done_keys = {(r["image_id"], r["concept"]) for r in data}` **unconditionally** from **all** records — it does not filter `raw_response` starting with `"ERROR:"`. The eval loop skips any key in `done_keys` (402). A pair that failed on a transient infra error (OOM, CUDA error, pipeline timeout) is therefore **permanently locked** as `"Uncertain"` and never re-judged — even though `--resume` exists precisely to recover from such failures. The error is indistinguishable from a genuine semantic `"Uncertain"` in the output CSV because `_save_final` strips `raw_response` (497-503).
- **Evidence:**
  ```python
  # 458-465
  records.append({...,"verdict":"Uncertain","raw_response":f"ERROR: {e}"})
  # 261-262
  return {(r["image_id"], r["concept"]) for r in data}   # no filter
  # 402
  if key in done_keys: continue
  ```
- **Impact:** Transient infra failures become permanent: the verdict is frozen at `"Uncertain"` and counted as genuine in `Score(c)`. `Score(Uncertain)` inflated, `Score(Aligned/Unaligned)` deflated — **irreversibly and with no signal in the CSV**. A crashed-then-resumed run yields a metric that is a mixture of real verdicts and frozen errors. No test exercises exception-then-resume (`test_resume_skips_done` only seeds a successful record).
- **Remediation:** Exclude errored records from `done_keys`: `{k for r in data if not str(r.get("raw_response","")).startswith("ERROR:")}`; **or** add a `status` field and filter on it; **or** skip appending to `records` on exception (count separately, let `--resume` retry). At minimum persist `raw_response` to the CSV so failures are auditable. Add `test_resume_retries_errored_pair`.

### 🟠 High (S1)

#### F-003 — Retry path samples at temperature=0.3 with NO seed; judge never calls utils.set_global_seed → non-reproducible verdicts
- **Severity:** S1 · **Category:** reproducibility · **Stage:** LLM generation
- **Location:** `src/evaluate_llm_judge.py:166-173` + entire module (no seed)
- **Agents:** all 9 · **Orchestrator-verified:** ✅ (grep)
- **Description:** First attempt is greedy (`do_sample = retries>0` → `False`), deterministic. But any pair whose greedy response fails `_extract_verdict` triggers a retry; on retry `generation_kwargs = {max_new_tokens:10, do_sample:True, temperature:0.3}` (170-173). No `torch.manual_seed` / generator / `set_global_seed` anywhere in the module (grep: 0 hits). The existing helper `utils.set_global_seed` (`utils.py:41-48`, seeds random/np/torch/cuda + `cudnn.deterministic`) is used by `train_sae.py` and `sae_module.py` but **never imported or called** by the judge. Spec mandates `temperature=0.0` deterministic (`implementation.md:190,235`).
- **Impact:** `Score(c)` is non-reproducible for any run touching the retry path: re-running on identical inputs/hardware can flip verdicts between a valid label and fallback `Uncertain`. Compounds with F-001 (inflates retry rate) and F-002 (first run's random retry verdicts get frozen). Breaks the "completely riproducibile" guarantee (`implementation.md:172`).
- **Remediation:** Call `utils.set_global_seed(config.training.primary_seed)` at the top of `evaluate()` (reuse existing helper). **And/or** keep `do_sample=False` on retries and rely on `RETRY_SUFFIX` (64) prompt variation to break greedy loops. If sampling retained, persist the seed per record.

#### F-004 — Headline Alignment Rate divides Aligned by ALL records including infra-error and fallback "Uncertain" → mixes infrastructure failures with semantics
- **Severity:** S1 (downgraded from S0) · **Category:** evaluation · **Stage:** metric reporting
- **Location:** `src/evaluate_llm_judge.py:508-531` (`_print_summary`)
- **Agents:** all 9 · **Orchestrator-verified:** ✅
- **Description:** `_print_summary` computes `total = len(df)` over **every** record (511), then `aligned_rate = counts.get("Aligned",0)/total` (530) and per-verdict `pct = 100*n/total` (526). `df` includes genuine verdicts, fallback `"Uncertain"` (exhausted retries, 209), **and** exception-handler `"Uncertain"` (infra failures, 458-465). The MedConcept `Score(c)` is defined over `v_i = LLM verdict`; an infra-failure/parse-fallback `"Uncertain"` is not a genuine verdict yet is counted in both denominator and numerator. *Downgraded to S1: the metric is only printed to stdout, never persisted to the CSV — blast radius is the console/log, not silent-wrong persisted data. But it is the single number a human reads as concept quality.*
- **Impact:** Reported Alignment Rate is systematically deflated by `(errors+fallbacks)/total`; Uncertain inflated by the same amount. Two runs with identical concept quality but different transient error rates report materially different headline metrics. The error count is printed separately (517) but **not** excluded from the denominator — inconsistent.
- **Remediation:** Compute `aligned_rate` (and per-verdict pct) over valid semantic verdicts only: `total_valid = total - errors - fallback_count` (track `fallback_count` separately, currently uncounted), or exclude records whose `raw_response` starts with `"ERROR:"`. Print both raw and error-adjusted rates.

### 🟡 Medium (S2)

#### F-005 — aligned_scores.csv has NO Score(c) column despite its name — the MedConcept metric is computed and printed but never persisted (contract/naming mismatch)
- `src/evaluate_llm_judge.py:483-505` (`_save_final`) + `508-533` (`_print_summary`) · all 9 agents.
- `_save_final` writes exactly `output_cols = [image_id, feature_id, concept, activation, verdict]` (498-501); no score column. `Score(c)` is computed only inside `_print_summary` as per-verdict percentages and `aligned_rate` (522-531) and sent to stdout via `print()` — never written to the CSV or any JSON sidecar. The filename `aligned_scores` implies a score is present but the file carries only raw per-pair verdicts. *(The spec's reference CSV `implementation.md:688` also writes verdicts only, so the schema matches the spec literally — but the formal metric is never materialized as an artifact.)* **Impact:** every downstream consumer must recompute `Score(c)`; nothing prevents a consumer computing it over a filtered/changed subset and silently disagreeing (e.g. including error rows per F-002/F-004). If stdout is lost, the score is gone. `test_full_pipeline` asserts exactly these 5 columns, locking in the absence. **Remediation:** write a companion `results/judge_scores.json` with `{aligned, unaligned, uncertain, aligned_rate, n_total, n_errors}`, or add a score column; at minimum document that the CSV is raw per-concept verdicts and `Score(c)` is derived.

#### F-006 — Images whose report has empty findings+impression are silently dropped from evaluation (coverage bias in Score denominator)
- `src/evaluate_llm_judge.py:348-354` (`_combine_report`) + `367-371` (`report_lookup`) + `388-392` (skip) · 7/9 agents.
- `_combine_report` joins `findings + impression`; if both NaN it returns `""`. When building `report_lookup` (367-371), `if report_text:` skips empty strings so the filename never enters the lookup. In the eval-pair loop (388-392), `report = report_lookup.get(image_id)` returns `None` → `if not report: skipped_no_report += 1; continue` drops the image **and all its top-k concepts**. The MedConcept `|P|` is "total predicted concepts" but here it is silently reduced to concepts from images with non-empty `findings`/`impression`. Only a warning count is logged (413); excluded pairs leave no trace in the CSV. **Impact:** coverage bias — if no-report images are not missing-at-random (e.g. normal studies with blank findings, the hardest Aligned/Unaligned cases), `Score(c)` is computed over a non-representative, easier subset. Magnitude unknown (drop count not surfaced). **Remediation:** persist `skipped_no_report` image_ids to `results/judge_coverage.json` as a first-class coverage statistic; either judge no-report images as `Uncertain` (count in `|P|`) or document the exclusion and report `exclusion_rate` alongside `Score(c)`.

## Metric Validation

`Score(c) = (1/|P|)·Σ 1(v_i = c)` is computed **correctly arithmeticly** inside `_print_summary` (per-verdict `value_counts / total`, line 526; `aligned_rate = Aligned/total`, line 530) — **but** computed over a **contaminated population** and **never persisted**. Three correctness problems:

1. **Denominator `|P|` = `len(df)` = all records**, including infra-error `Uncertain` (F-002) and fallback `Uncertain` (F-001/F-004) — not the set of genuine LLM verdicts the formula intends. `Score(Aligned)` deflated, `Score(Uncertain)` inflated by the error/fallback fraction.
2. **`|P|` silently excludes** all concepts from no-report images (F-006) — a non-random subsample.
3. **`Score(c)` is only printed** to stdout, never written to `aligned_scores.csv` or any sidecar (F-005) — no canonical persisted value; the verdict vector feeding it is itself biased by F-001.

**Verdict: metric formula correct; metric population + persistence incorrect.**

## Data Integrity

The `filename → uid → report` join is **sound** (verified 335-371). Two integrity concerns:

1. **No-report images silently dropped** (F-006) — coverage is not 100% and not surfaced as a metric, so `|P|` is a biased subsample.
2. **Errored pairs indistinguishable from genuine verdicts** (F-002) — persisted as `verdict="Uncertain"` with `raw_response="ERROR:…"`, included in `done_keys` and in the output CSV (after `raw_response` is stripped). A downstream consumer cannot tell infra failures from real verdicts.

Both corrupt the integrity of the verdict set feeding `Score(c)`.

## Reproducibility

**Non-deterministic and not resumable-cleanly.**

1. **No seed** anywhere in `evaluate_llm_judge.py` (grep-confirmed; `utils.set_global_seed` exists at `utils.py:41` but is never called); retry path uses `do_sample=True, temperature=0.3` with no seed/generator (F-003). Spec mandates `temperature=0.0`.
2. **Resume unsound for errored pairs:** `--resume` treats errored pairs as done (F-002) → a crashed-then-resumed run freezes transient errors as permanent `Uncertain` verdicts. The resumed state is not a faithful re-run.
3. **`bfloat16` CUDA inference + `device_map="auto"`** (line 89) add further nondeterminism not controlled by any seed.

The judge stage does **not** meet the reproducibility standard the rest of the pipeline (`train_sae`, `sae_module`) sets via `utils.set_global_seed`.
