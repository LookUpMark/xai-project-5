**One-sentence verdict:** The implementation is mechanically correct, but the core methodological choice — training a *seed-dependent* Top-K SAE on the *512-d projected* BiomedCLIP embedding with only ~6k images — fights a structurally non-identifiable sparse-decomposition problem in an anisotropic, information-bottlenecked space; the reported failure (cross-seed Jaccard at the analytical chance floor, 40-60 % dead features, SAE barely beating random on concept naming) is the *expected* symptom of that setup, not a bug.

This audit identifies **3 remediation paths** (§6), of which **Path A (move the SAE to the 768-d pre-projection hidden state)** and **Path B (pivot to pure SPLiCE-style decomposition)** are the high-leverage options; Path C (tune the existing 512-d SAE) is unlikely to move the headline metric on its own.

## 1. The Central Question — Which Representation?

The user's explicit question: *"Are the embeddings used to train the SAE the right ones, or should we have used the previous layer of the VLM encoder?"*

### 1.1 What the code actually extracts

BiomedCLIP (`chuhac/BiomedCLIP-vit-bert-hf`) is a ViT-B/16 vision encoder (hidden dim **768**) followed by a **linear projection to 512** (the shared image-text space). The pipeline extracts the **post-projection** output:

```python
# src/embedding_extraction/extract_embeddings.py:63-64
outputs = model.get_image_features(**inputs)          # (B, 512)  — POST-projection
outputs = outputs / outputs.norm(dim=-1, keepdim=True) # L2-normalised
```

So the SAE trains on the **512-d L2-normalised projected embedding**. SAE input dim = 512 (`config.py:260`, `activation_dim=512`). This is mechanically correct and L2-norm matches the literature convention.

### 1.2 The literature is genuinely *split* into two paradigms

This is the crux, and it was mis-stated in early exploration. Verified facts:

| Paradigm | Operates on | Naming mechanism | Canonical method |
|---|---|---|---|
| **A — Concept discovery / bottleneck** | **final projected embedding** (shared img-text space) | *direct*: cosine of decomposition coefficients vs concept-text embeddings | **SPLiCE** (Bhalla et al., NeurIPS 2024) |
| **B — SAE interpretability** | **intermediate residual-stream / MLP hidden state** (768) | *separate* (probe, steering, sparse-text projection) | Steering CLIP's ViT with SAEs (arXiv 2504.08729); OpenAI Scaling SAEs |

**Paradigm A — SPLiCE (verified):** SPLiCE decomposes the **final CLIP image embedding** into a *sparse, non-negative linear combination of concept dictionary atoms*, where the dictionary atoms are **text embeddings of human-interpretable concepts living in the same shared 512-d space**. It is explicitly a post-hoc method on the projected embedding — it does **not** use hidden states. *(Sources: [NeurIPS 2024 proceedings](https://proceedings.neurips.cc/paper_files/paper/2024/hash/996bef37d8a638f37bdfcac2789e835d-Abstract-Conference.html), [arXiv 2402.10376](https://arxiv.org/abs/2402.10376), [github.com/ai4life-group/splice](https://github.com/ai4life-group/splice).)*

**Paradigm B — SAE-on-CLIP interpretability (verified):** the line of work that trains *Top-K / JumpReLU SAEs* on CLIP does so on **residual-stream activations of (each) ViT layer**, not on the final projection. The final projection is regarded as a lossy bottleneck unsuitable for sparse feature discovery. *(Sources: [Steering CLIP's ViT with SAEs, arXiv 2504.08729](https://arxiv.org/html/2504.08729v1); [clip-topk-sae, HF lasgroup](https://huggingface.co/lasgroup/clip-topk-sae); [Scaling and Evaluating SAEs, OpenAI](https://cdn.openai.com/papers/sparse-autoencoders.pdf).)*

### 1.3 Why this matters — the project mixes the two paradigms

The project uses:
- the **tool** of Paradigm B (a *seed-dependent*, learned Top-K SAE),
- on the **location** of Paradigm A (the 512-d *projected* embedding),
- with the **naming** of Paradigm A (cosine of decoder rows vs RadLex text embeddings, `concept_naming.py`).

This combination inherits the **worst of both**:

1. The 512-d projected space is **anisotropic**, **modality-gap-shifted**, and designed for text-image *alignment*, not for preserving sparse-linear structure. A learned SAE must factor a matrix that has already been through a lossy, regularising projection.
2. Sparse matrix factorisation `X ≈ D·A` is **non-identifiable** without sufficient data; seed-to-seed the optimum is degenerate. This is *exactly* the failure mode the reports observe (Jaccard ≈ analytical null).
3. Naming-by-cosine *requires* the decoder rows to live in the text-aligned space — so if one were to "fix" the representation by moving to 768-d hidden states (Paradigm B), the current naming bridge **breaks** (RadLex text embeddings live in the 512-d shared space). This is the real trade-off, and it is not free.

**Finding M-001 (🔴 Critical, methodological):** *The SAE is trained on the wrong representation for the tool being used.* Either (a) keep the projected embedding and drop the learned SAE in favour of SPLiCE-style direct decomposition against a fixed RadLex dictionary (Path B), or (b) keep the SAE but move it to the 768-d pre-projection hidden state and build a separate concept-naming bridge (Path A). Training a learned, seed-dependent SAE in the 512-d projected space combines an underdetermined target with a seed-dependent solver.

> **Note on confidence:** the *direction* (SAE-interpretability works on hidden states; SPLiCE works on the projected embedding) is verified from primary sources. The claim that moving to 768-d *will* fix the instability is a **hypothesis**, currently the team's own ablation #1 (`ADDITIONAL-ABLATION-STUDIES.md`) — listed but **never run**. It is the highest-leverage untested experiment in the project.

## 2. Findings (Methodological)

> Severity: 🔴 Critical · 🟠 High · 🟡 Medium · 🟢 Low. IDs `M-xxx` = methodological (this audit); `F-xxx` = code bugs (prior audits, referenced not re-litigated).

### M-001 🔴 SAE trained on 512-d projected embedding, not hidden state
See §1. The representation choice is inconsistent with the SAE-interpretability literature and is the root cause of the structural instability. **Highest-leverage untested fix.**

### M-002 🟠 Dataset is 2-3 orders of magnitude too small for a *learned* SAE
- **5,976 train images vs 4,096 dictionary features ≈ 1.8 samples/feature.** SAE/dictionary-learning practice and the SPLiCE/SAE-CLIP literature operate at ImageNet-to-LAION scale (10⁵-10⁶+ samples). The project is ~10²-10³ below.
- Small data + overcomplete dictionary ⇒ high dead-feature rate and non-identifiable factorisation. The observed **40-60 % dead features** and **Jaccard at the analytical chance floor** are the textbook symptom.
- **Note on domain fit:** BiomedCLIP is *not* the wrong backbone — it is domain-pretrained on biomedical text/images. The problem is **scale, not domain mismatch**. IU X-Ray (7,470 images) is simply small.

### M-003 🟠 Data augmentation, as implemented, *harms* the dictionary
Augmentation (3× expansion) was added to mitigate M-002. Results show it **backfired**:
- Dead features **increased +15-19 pp** (standard → augmented) across dict sizes.
- Dictionary utilisation **dropped 8-19 pp**.
- Cross-seed Jaccard **unchanged** at the chance floor (ratio ~1.0).
- JumpReLU stability **collapsed 290×** (0.0081 → 0.000028).

**Why:** horizontal flip / mild crop / rotation on chest X-rays produces **near-duplicate BiomedCLIP embeddings** (the encoder is approximately invariant to them). Adding 3× near-duplicates does not add new directions in the 512-d manifold — it *concentrates* activation mass onto already-popular features and starves rare-direction features, raising the dead count. This is the **smoking gun** that the bottleneck is the *effective diversity of directions in the projected space*, not the raw sample count. Naive augmentation cannot rescue a non-identifiability problem.

### M-004 🟡 Reconstruction cosine is uninformative (saturated)
Reconstruction cosine is 0.98-0.998 **across every ablation and both settings**. With a 512-d L2-normalised input, a low-dimensional manifold, and an 8×-overcomplete dictionary with k=32, near-perfect reconstruction is trivially achievable and carries ~no signal about feature quality. **Reconstruction should not be reported as evidence the SAE "works".** The reports are mostly careful about this, but the metric's presence in headline tables risks misreading.

### M-005 🟡 Concept naming barely beats random — FreqKMeans dominates
- SAE naming mean ≈ **0.395**, random baseline ≈ **0.372**, dense-PCA ≈ 0.380 — **SAE is only +0.023 above random.**
- FreqKMeans reaches **0.834** — the SAE is crushed by a trivial clustering baseline on the naming axis.

This is the most damaging result for the project's thesis ("SAE discovers interpretable concepts"): the learned SAE features are **not measurably more semantically aligned** to RadLex than random directions. Consistent with M-001/M-002 — if the factorisation is non-identifiable, the decoder directions are noise, so their cosine-to-vocab is noise-level.

### M-006 🟡 Hyperparameters are mis-sized for a small dataset
- `n_training_steps = 50,000`, `batch_size = 256` ⇒ ~12.8M samples seen ≈ **~2,140 epochs** over 5,976 embeddings. Severe overfitting risk on a degenerate solution.
- Auto-scaled LR ≈ **4e-4** (from `2e-4/√(4096/16384)`). The team's own `SAE-TRAINING-SMALL-DATASET.md` recommends **5e-5** for small data — an order of magnitude lower.
- `dict_size = 4096` (8× of 512). Ablation 01 already showed shrinking to 1024/2048 *reduces dead features but does not lift Jaccard* — confirming the problem is the space/data (M-001/M-002), not the width.

### M-007 🟢 LLM-judge faithfulness numbers are not yet trustworthy
The 06-24 audit documents 🔴 bugs in `evaluate_llm_judge.py` (verb/label mismatch F-001, frozen-on-resume F-002, non-reproducible temperature F-003, no `JudgeConfig` F-007). Until those are fixed, any faithfulness score produced by the judge is biased. **Out of scope here** — tracked in `ML-AUDIT-2026-06-24.md`. Flagged only because "faithfulness" appears in ablation 05.

### M-008 🟢 Cross-seed Jaccard interpretation is correct
The headline **Jaccard 0.0038 ≈ analytical null k/(2D−k) = 0.0039** (ratio ~0.97-1.00) is correctly read by the reports as "at the mathematical chance floor" — confirmed independently by A00 (direction-space consensus also ≈ 0, ruling out a permutation artifact) and A03 (signal-to-null ~0.95). This is **not** a measurement error; it is a real null result. **PASS.**

## 6. Remediation Paths (implementation steps)

Three paths, ranked by expected leverage. Paths A and B attack M-001 directly; Path C does not (and is expected to underdeliver on the headline metric).

### Path A — Keep the SAE, move it to the 768-d pre-projection hidden state (🔴 addresses M-001)

Attacks the root cause. Most aligned with the SAE-interpretability literature.

1. **New extraction mode.** In `src/embedding_extraction/extract_embeddings.py`, add a branch that returns the pre-projection activation instead of `get_image_features()`:
   ```python
   # CLS token of the last hidden state (768-d), pre-projection
   out = model.vision_model(pixel_values=pixel_values, output_hidden_states=True)
   hidden = out.last_hidden_state[:, 0, :]          # (B, 768)
   # (Optional: ablate a deep residual layer instead — out.hidden_states[-2])
   ```
   Verify the exact HF attribute on `chuhac/BiomedCLIP-vit-bert-hf` before relying on it.
2. **Config.** `EmbeddingConfig` + `SAEConfig.activation_dim: 768`; reconsider `dict_size` (8× of 768 = 6144, or keep 4096 ≈ 5.3×). Keep `k=32`, lower `steps` and `lr` per Path C defaults.
3. **Persist to a new dir** (`embeddings/standard_hidden/`, `augmented_hidden/`) so results stay comparable to the 512-d baseline. Regenerate `modality_gap.pt` for the new space.
4. **Retrain 5 seeds** (`python src/autoencoder/train_sae.py`).
5. **Build a naming bridge.** Decoder rows now live in 768-d, RadLex text embeddings in 512-d. Options: (a) train a linear probe `768 → 512` from the frozen BiomedCLIP projection, then cosine-match; (b) compute sparse-text projection à la the SAE-CLIP literature. Update `concept_naming.py` accordingly.
6. **Re-run** `concept_naming.py`, `stability_analysis.py`, `generate_explanations.py`.
7. **Verify:** cross-seed Jaccard lifts materially above `k/(2D−k)`; dead-feature rate drops; naming mean rises above the random baseline by a clear margin.

**Files touched:** `extract_embeddings.py`, `notebooks/vlm/extract_embeddings.ipynb`, `config.py`, `train_sae.py`, `sae_module.py`, `concept_naming.py`.

### Path B — Pivot to pure SPLiCE-style decomposition (no learned SAE) (🔴 addresses M-001 + robust to M-002)

Drops the seed-dependent learned dictionary entirely. The dictionary is **fixed** = RadLex text embeddings; per image, solve a sparse non-negative regression of the 512-d image embedding against the dictionary. Deterministic ⇒ no seeds, no Jaccard instability; naming is intrinsic (the coefficients *are* the concepts). More robust on small data because the dictionary is not learned from the 6k samples.

1. **New module** `src/concept_discovery/spliece.py`: per-image sparse non-negative least-squares / L1-regularised regression of `image_emb (512)` on `vocab_emb (RadLex, 512)`. Use `sklearn.linear_model` (Lasso with `positive=True`) or `scipy.optimize.nnls` with a sparsity budget. Reuse the existing `vocabulary.json` + `text_vocab_embeddings.pt`.
2. **Apply modality-gap correction** (already computed) before regression — same trick the SAE naming uses.
3. **Naming = direct.** Top-k vocabulary terms by coefficient per image; global concept ranking by activation frequency.
4. **Stability** is trivial (deterministic); **faithfulness** runs through the existing judge pipeline (after M-007 fixes).
5. **Verify:** concept coherence on sample images; faithfulness ≥ SAE baseline; compare concept-coverage of RadLex vs the SAE's live features.

**Files touched:** new `src/concept_discovery/spliece.py`, `concept_naming.py` (or a parallel path), new report notebook under `notebooks/autoencoder/`.

### Path C — Tune the existing 512-d SAE (🟡 addresses M-006 only; unlikely to fix M-001)

Expected to improve dead features / reconstruction but **not** to lift Jaccard off the floor (Ablations 01/02 already showed width and k do not move it). Include for completeness and as a cheap companion to A/B.

1. **Reduce epochs:** `n_training_steps` 50,000 → ~5,000-10,000 (≈ 200-400 epochs).
2. **Lower LR:** set explicit `lr = 5e-5` (per `SAE-TRAINING-SMALL-DATASET.md`) instead of the auto ~4e-4.
3. **Shared initialisation / model soup across seeds** (Ablation A03): initialise seeds 1..n from seed-0 weights, or average the decoder matrices post-hoc.
4. **Dict size** 4096 → 2048 (reduces dead %; will not fix Jaccard).
5. **Dead-feature resampling** if the `dictionary-learning` library exposes it for `AutoEncoderTopK`.
6. **Augmentation rework (M-003):** current augmentations create near-duplicates. Either (a) drop augmentation for the SAE stage, or (b) use intensity/stronger-geometric transforms that actually move the BiomedCLIP embedding, or (c) accept that augmentation does not help in the projected space and document it.
7. **Verify:** dead %, reconstruction, Jaccard. **Honest expectation:** Jaccard stays near the floor; this path alone is insufficient.

**Files touched:** `config.py` (`SAEConfig`, `TrainingConfig`, `AugmentationConfig`), `train_sae.py`, `src/augmentation/transforms.py`.

### Recommended sequencing

1. **Path A first** (decisive test of M-001). If Jaccard lifts → the representation was the problem; proceed to productionise 768-d + naming bridge.
2. **Path B in parallel or as fallback** (robust-by-construction; insures against A not working).
3. **Path C** as cheap hygiene regardless (epochs/LR are wrong for the data size and should be fixed in any case).
4. **Path M-007 (judge fixes)** is a prerequisite for trusting any faithfulness number — sequence independently per the 06-24 audit fix-plan.

## 7. Open Questions

1. **Does Path A actually lift Jaccard?** Hypothesised yes, untested. Highest-priority experiment.
2. **Naming bridge for 768-d (Path A step 5):** which probe — frozen projection, learned linear, or sparse-text? Needs a small design decision before implementation.
3. **Is the RadLex vocabulary (≈508 terms) the right concept space?** It is generic-radiology; `VOCAB-BUILDING-ALTERNATIVES.md` notes MeSH/Problems (~118 in-distribution terms) as alternatives. Vocab quality caps naming scores regardless of representation.
4. **Can IU X-Ray ever support a learned SAE?** Even at 768-d, 6k images is small. Path B (fixed dictionary) is the more data-frugal method if A underdelivers.
